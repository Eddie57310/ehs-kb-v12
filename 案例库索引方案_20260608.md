# 案例库索引方案（is_index 机制）

**整理日期**：2026-06-08  
**适用系统**：EHS 知识库 V11

---

## 一、背景与问题

### 1.1 核心问题

知识库里的案例类文件（如排雷库、惊喜库）在入库时按页或按段落切块，每个 chunk 只是一条案例的局部内容。当用户问"有多少案例"或"有哪些案例"时，系统行为如下：

1. 向量检索命中若干案例卡片片段（如 5～8 条）
2. LLM 数了上下文里出现的案例片段，回答"共 5 条"
3. 实际上库里有 123 条，答案严重错误

**根本原因**：向量检索是"相似度匹配"，只会返回最相关的几个片段，不会返回全部内容。LLM 没有"扫描全库计数"的能力，只能基于上下文回答。

### 1.2 为什么不直接统计 ChromaDB

理论上可以用代码统计 `where={"source": "xxx"}` 的 chunk 数，但：
- chunk 数 ≠ 案例数（一条案例可能跨多个 chunk）
- 无法处理"按专业分多少条"这类结构化问题
- 需要额外接口，增加系统复杂度

### 1.3 方案选择

**人工维护一份库索引 MD 文件**，把案例的总数、分类、明细列表写清楚，入库时标记为索引文件，检索时整份带入上下文，由 LLM 直接读数。

---

## 二、方案设计

### 2.1 整体流程

```
用户提问（"有多少案例"/"有哪些案例"）
        ↓
混合检索（向量 + BM25）
        ↓
命中某个 is_index 文件的任意 chunk
        ↓
【索引聚合】：从 ChromaDB 拉出该文件全部 chunk，按 chunk_seq 排序
        ↓
整份索引置于上下文最前面
        ↓
LLM 收到 index_hint 指令："从索引文件的明细列表中统计"
        ↓
LLM 直接读索引，给出准确数字
```

### 2.2 三个关键组件

| 组件 | 位置 | 作用 |
|------|------|------|
| `is_index` 标记 | ChromaDB chunk metadata | 标识该 chunk 来自索引文件 |
| 索引聚合逻辑 | `feishu_ws_server_v11.py:982` | 命中即整份带出 |
| `index_hint` 指令 | user_prompt（问题后面） | 告诉 LLM 从索引统计 |

## 三、is_index 标记：如何识别索引文件

### 3.1 识别规则（`index_reviewed_md.py:158`）

```python
def _is_index_file(rel_path: str) -> bool:
    kb_dir = os.path.join(KB_DIR, os.path.dirname(rel_path))
    if not os.path.isdir(kb_dir):
        return False
    return any(
        d.endswith("案例") and os.path.isdir(os.path.join(kb_dir, d))
        for d in os.listdir(kb_dir)
    )
```

**规则**：一个 MD 文件，若其所在目录（在 `Local_KB/` 中）下存在名称以"**案例**"结尾的子目录，则该文件标记为 `is_index=True`。

**示意图**：
```
Local_KB/
├── 客关部/
│   ├── 排雷库/
│   │   ├── 20260401_排雷库清单.md   ← ✅ is_index=True
│   │   │                              （同级有"排雷库案例"子目录）
│   │   └── 排雷库案例/
│   │       └── xxx.pdf
│   └── 规定/
│       └── xxx.md                   ← ❌ 同级无"案例"目录
│
└── 设计部/
    ├── 20260608_设计部案例清单.md   ← ✅ is_index=True
    │                                  （同级有"案例"子目录）
    └── 案例/
        └── 惊喜库v1.pdf
```

### 3.2 入库时打标（`index_reviewed_md.py:191`）

```python
metadata = {
    "source":    rel_path,
    "timestamp": timestamp,
    "date_str":  date_str,
    "category":  category,
    "domain":    domain,
}
if _is_index_file(rel_path):
    metadata["is_index"] = True

chunks = _structured_chunks(md_content)
docs = [
    Document(page_content=chunk, metadata={**metadata, "chunk_seq": i})
    for i, chunk in enumerate(chunks, start=1)
]
```

每个 chunk 都带 `chunk_seq`（从 1 开始的序号），聚合时用于还原文件顺序。

## 四、索引聚合逻辑（检索时）

**触发条件**：Reranker 精排后的 `valid_with_scores` 中，有任意一条 chunk 的 `is_index=True`。

**代码**（`feishu_ws_server_v11.py:982`）：

```python
# 1. 扫描命中结果，收集所有 is_index 文件的 source
index_srcs = {}
for doc, score in valid_with_scores:
    if doc.metadata.get('is_index'):
        src = doc.metadata.get('source')
        if src not in index_srcs:
            index_srcs[src] = score

if index_srcs:
    # 2. 把命中结果中 is_index 的片段去掉（避免重复）
    kept = [(d, s) for d, s in valid_with_scores if not d.metadata.get('is_index')]

    # 3. 从 ChromaDB 拉出整份索引文件，按 chunk_seq 排序
    aggregated = []
    for src, score in index_srcs.items():
        full = db.get(where={'source': src}, include=['documents', 'metadatas'])
        blocks = sorted(zip(full['documents'], full['metadatas']),
                        key=lambda x: x[1].get('chunk_seq', 0))
        for content, meta in blocks:
            aggregated.append((_LCDoc(page_content=content, metadata=meta), score))

    # 4. 整份索引置于上下文最前面，其余命中块跟后
    valid_with_scores = aggregated + kept
```

**效果**：上下文结构变为：
```
[参考资料 1～18]  ← 索引文件全部 18 个 chunk（总览+各版本明细）
[参考资料 19～N]  ← 其他检索命中的案例卡片片段
```

## 五、index_hint 指令

当 `has_index_aggregation=True` 时，在 user_prompt 的问题后面（LLM 指令区域）插入：

```python
index_hint = (
    '【重要】参考资料中含有本库的索引/清单文件（标题含"总览""清单""介绍"等），'
    '其中列出了各系列/版本的案例或条目明细。'
    '回答数量或枚举类问题时，请从该索引的明细列表中统计，'
    '不要仅凭正文案例卡片片段来估算总数。\n'
) if has_index_aggregation else ""
```

**为什么放在问题后面而不是上下文前面**：  
LLM 对 prompt 结构的权重不同。放在"参考资料"区域会被当作普通文本，权重低；放在"指令区域"（问题后面、格式要求前面）权重最高，LLM 会优先遵守。

**演进历史**：
- v1（已废弃）：硬编码"排雷库清单总览"和"共收录123条问题"字样。设计部查询时 LLM 找不到匹配，回答"未在知识库中找到"。
- v2（当前）：通用描述，不提具体库名或数字，适用所有库。

## 六、索引 MD 文件规范

### 6.1 现有索引文件

| 文件 | 路径 | 覆盖内容 |
|------|------|---------|
| 排雷库清单 | `客关部/排雷库/20260401_排雷库清单.md` | 123条问题，按风险类型、专业、阶段分类 |
| 设计部案例清单 | `设计部/20260608_设计部案例清单.md` | 6个文件总览（134条惊喜库卡片+用心库+沧海桑田） |

### 6.2 索引文件结构规范

以排雷库清单为参考模板，一份好的索引文件应包含：

```markdown
# [库名]介绍
（1-2句话说明库的定位、来源、保密级别）

# [库名]总览
（一张汇总表，含：总条数、分类维度、各维度数量）

# [维度1] 分类明细
（逐条列出，格式：编号 + 案例名/条目名）

# [维度2] 分类明细
...
```

**关键要求**：
1. **总览表必须写明总条数**（如"共收录 123 条"），是 LLM 的权威数字来源
2. **各维度的明细列表必须完整**，LLM 可从中逐条数
3. 标题使用 `#`/`##`/`###` 层级，`_structured_chunks` 会按标题切块
4. 文件名含日期前缀（`YYYYMMDD_`）

### 6.3 切块行为

`index_reviewed_md.py` 中的 `_structured_chunks` 按 `##` 二级标题切块：
- 每个二级标题下的内容为一个 chunk
- 每个 chunk 带上父标题作为前缀，保留层级语义
- 18 个 chunk 是设计部案例清单的实际切块数量

切块越细（每个版本/维度一个 chunk），聚合后 LLM 读到的结构越清晰。

## 七、局限性与待改进

### 7.1 人工维护负担

索引文件的条目数量是**手动维护**的，每次案例更新需同步修改：
- 总览表的数字
- 对应版本的明细列表

如果漏改总览表，LLM 给的数字就会过期。

**后续优化方向**：写脚本扫描案例 PDF 的 chunk，自动统计并更新索引 MD 中的数字。

### 7.2 is_index 识别规则的边界

当前规则"同级有以'案例'结尾的子目录"比较简单，可能误判：
- 某目录下既有案例子目录，又有多个 MD 文件 → 每个 MD 都会被标记，只应有一个是索引
- 未来若出现"案例库"子目录（4字），同样会触发

目前实际使用中未出现问题，因为每个库目录下通常只有一个对应的清单 MD。

### 7.3 聚合 chunk 数量限制

整份索引文件（如 18 个 chunk）全部置入上下文，会占用 LLM 的 token。如果索引文件过大（如超过 50 个 chunk），可能撑满 context window，挤掉其他检索结果。

当前设计部案例清单 18 块 × 平均约 300 字 ≈ 5400 字，可接受。

### 7.4 仅适用于人工维护的 MD 索引

设计部案例类（PDF）直接入库，没有 MD 中间文件。用心库、沧海桑田计划的详细条目数目前在 MD 清单里有"（请补充）"占位，LLM 对这两个库的条目数还无法准确回答。

## 八、操作流程（新增一个库的索引）

1. **在对应库目录下创建清单 MD 文件**  
   命名：`Local_KB/<部门>/<库名>/YYYYMMDD_<库名>清单.md`  
   内容参考第六节规范。

2. **确认目录结构触发 is_index**  
   确保 MD 文件所在目录下存在以"案例"结尾的子目录：  
   ```
   Local_KB/<部门>/<库名>/
   ├── YYYYMMDD_<库名>清单.md   ← 这个文件
   └── <库名>案例/              ← 这个目录的存在触发 is_index
   ```

3. **复制到 reviewed_md 并入库**  
   ```bash
   cp Local_KB/<部门>/<库名>/xxx清单.md \
      reviewed_md/<部门>/<库名>/xxx清单.md
   venv/bin/python index_reviewed_md.py
   ```

4. **验证**  
   ```bash
   python3 -c "
   import chromadb
   c = chromadb.PersistentClient(path='chroma_db')
   col = c.get_collection('langchain')
   r = col.get(where={'is_index': True})
   for s in set(m['source'] for m in r['metadatas']):
       print(s)
   "
   ```
   应能看到新增的清单文件 source。

5. **测试**  
   在飞书问"[部门] [库名]有多少案例"，确认数字与清单总览一致。
