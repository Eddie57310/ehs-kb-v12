# EHS 知识库 源文件编制要求

> 本文件规定向 EHS 智慧助手知识库上传的源文件编制标准，旨在确保文件能被系统准确识别、解析和入库。发现新问题后持续更新。

---

## 1. 表格规范

### 1.1 禁止使用合并单元格

**不符合要求：**

| 风险等级 | 单位名称 |
|---|---|
| 高风险（合并行） | 总承包单位 |
| （合并，留空） | 幕墙单位 |

**符合要求：**

| 风险等级 | 单位名称 |
|---|---|
| 高风险 | 总承包单位 |
| 高风险 | 幕墙单位 |

> 原因：知识库解析系统不支持合并单元格，跨行/跨列合并将导致数据错位或丢失，影响检索准确性。
>
> **操作要求：** 含表格的源文件上传前，须先取消所有合并单元格，取消后产生的空白单元格必须补充完整（将合并前的内容在每行中重复填写）。

---

### 1.2 禁止以图片形式呈现表格

**不符合要求：** 将表格截图或扫描后以图片形式插入文档。

**符合要求：** 使用 Word / Excel 原生表格功能制作，确保文字可复制选中。

> 原因：图片形式的表格无法被系统直接提取文字，即使经过 OCR 识别，大量数据的识别准确率也明显下降，导致入库数据残缺或错误。

---

## 2. 排雷库文件规范

### 2.1 排雷节点字段格式

**不符合要求：**
```
排雷节点  规划方案评审
```

**符合要求：**
```
排雷节点：规划方案评审
```

> **操作要求：** "排雷节点"后面须加冒号（中文冒号"："），保持字段格式统一，便于系统结构化解析。

---

### 2.2 图片须包含问题描述

**背景：** 飞书聊天不支持"文本 + 图片 + 文本"的富文本交替排版，系统会先输出全部文字，再统一发送图片。因此用户看到图片时，上下文可能已经滚过，无法直接对应具体问题。

**不符合要求：** 图片中仅有现场照片，无任何文字说明。

**符合要求（任选其一）：**
- 在图片上直接标注简要问题描述（如加文字水印/标题栏）
- 在源文件中将问题描述文字紧贴图片上方，截图时一起截入

> **目标：** 单独看一张图片，就能看出它对应的是什么问题。

---

### 2.3 库目录结构（清单与案例分离）

每个"库"作为一个独立文件夹组织：库根目录只放索引清单，案例原件单独放入以"案例"结尾的子文件夹。

**符合要求：**
```
客关部/
└── 排雷库/                       ← 库文件夹
    ├── 20260401_排雷库清单.md     ← 索引清单（库根，唯一）
    └── 排雷库案例/                ← 案例原件子文件夹（须以"案例"结尾）
        ├── …排雷库V1.0.pdf
        ├── …排雷库V2.0.pdf
        └── …排雷库V3.0.pdf
```

**不符合要求：** 清单与案例 PDF 混放在同一目录、或案例直接堆在专业根目录下。

> **操作要求：**
> - 库根目录**有且仅有一个**索引清单 `.md`
> - 案例原件全部放进名为"<库名>案例"的子文件夹（目录名须以"案例"二字结尾）
>
> **原因：** 系统据此自动识别——库根清单 .md 会被标记为索引并在检索时整份返回；"*案例"子目录内的文件不会被当作清单重复入库。

---

### 2.4 清单索引文件编制

索引清单集中承载：库的介绍说明 + 完整条目清单（各维度/分项、编号、名称、条数）。案例卡片中**不再重复**清单与说明内容。

> **操作要求：**
> - 清单内含：总览（各维度条数表）、各维度/分项明细（编号 + 名称）、节点与分类等说明
> - 用 Markdown 标题（`#` / `##`）组织层级；表格遵守 1.1（无合并单元格）且前后留空行
> - 一个库的"共多少条 / 各维度条数"只在清单里写一处，**不要在多个文件里各写一份**
>
> **原因：** 检索命中清单任一处时系统按结构整份返回，"共多少条""罗列全部"这类问题才能召全且不自相矛盾；清单与案例职责分离，避免同一信息多版本并存导致冲突。

---

### 2.5 案例卡片字段模板（统一格式）

每张排雷卡作为一个独立案例编制，字段齐全、格式统一，每个字段"名称：值"：

```
编号：GQ-DX010
案例名称：……
排雷节点：……
问题分类：……
风险描述：……
风险原因：……
预防建议：……
涉及专业：……
涉及项目：……
产生后果：……
（问题描述文字 + 截图，紧随其后）
```

> **操作要求：**
> - 字段名后统一用中文冒号"："（见 2.1）
> - 编号严格按"维度-分项+数字"，如 `GQ-DX010`；**勿多写 0**（写成 `GQ-DX0010` 会与清单对不上号）
> - 一张卡 = 一个案例，不混入其他案例、清单页或目录
> - 图片遵守 2.2（含问题描述）
>
> **原因：** 统一字段结构利于解析与检索精排；编号规范确保案例与清单索引交叉引用一致。
