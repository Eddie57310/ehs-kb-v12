## 一、系统架构概览

本系统是基于飞书机器人的企业知识库问答系统，核心技术栈：

| 组件 | 技术 |
|------|------|
| 向量数据库 | ChromaDB（持久化，`chroma_db/`） |
| Embedding 模型 | BAAI/bge-m3（本地 CPU 推理） |
| BM25 关键词索引 | 自建，序列化为 `bm25_index.pkl` |
| Reranker | bge-reranker-v2-m3（本地 CPU） |
| LLM（问答） | 豆包 doubao-seed-2.0-pro（云端） |
| 机器人接入 | 飞书 WebSocket 长连接 |
| PDF 解析 | MinerU（CPU 模式）+ PyMuPDF (fitz) |
| PPTX 解析 | LibreOffice headless 转 PDF → PyMuPDF |

## 三、数据入库流程

### 3.1 常规文档（公司内部 / 国家规定 / 工管部）

```
1. 将文件放入 Local_KB/<对应目录>/
2. venv/bin/python rebuild_md.py          → MinerU 解析 PDF/Word → 生成 reviewed_md/
3. 启动 md_reviewer（端口 8084）人工校核 MD，插入图片、删除目录页、修复表格
4. venv/bin/python index_reviewed_md.py   → 切块 → Embedding → 入库 ChromaDB
```

**注意事项**：
- 步骤 2 和步骤 4 之间**必须人工审核**，不可自动连跑
- 目录索引页须在审核时删掉（对检索有害）
- Markdown 合并单元格须手动展开（重复填写合并项）
- 表格修复工具：`~/doc_parser_v11/table_fixer.html`

### 3.2 EHS 案例（Vision LLM 路线）

```
1. 将 PDF 放入 Local_KB/EHS案例/
2. venv/bin/python sync_kb_v11.py --domain EHS案例
   → 逐页 Vision LLM 识别 → 直接入库（无 MD 中间文件）
```

### 3.3 设计部案例（截图 + 文字提取）

```
1. 将 PDF 放入 Local_KB/设计部/案例/
2. venv/bin/python reindex_design_cases.py
   → PyMuPDF 提取文字 + 整页截图（144 DPI）→ 入库
   → 自动备份 chroma_db → 删旧 chunk → 入新 chunk → 重建 BM25
```

PPTX 文件质量差，已全部转换为 PDF 再入库（LibreOffice headless 转换）。

### 3.4 增量入库规则

`index_reviewed_md.py` 有增量逻辑：已入库的 `source` 路径自动跳过，重跑安全。  
如需强制重入某文件，需先从 ChromaDB 按 `source` 删除旧 chunk，再运行脚本。

## 四、检索流程

### 4.1 混合检索（Hybrid Retrieval）

```
用户提问
   ↓
ChromaDB 向量检索（k=20，bge-m3 embedding）
+ BM25 关键词检索（k=20，jieba 分词）
   ↓
RRF 互惠排名融合（Reciprocal Rank Fusion）
   ↓
Domain / 封闭库过滤
   ↓
Reranker 精排（bge-reranker-v2-m3，top_k=8）
   ↓
LLM 生成答案（豆包 doubao-seed-2.0-pro）
```

### 4.2 Domain 路由规则

提问中带关键词时：
- 检测到 `EHS案例` / `公司内部` / `国家规定` → 仅在该 domain 内检索
- 检测到 `客关部` / `设计部` / `工管部` / `合约部` → 仅在该封闭库内检索
- 未检测到任何关键词 → 在三大公开库中检索，**封闭库自动排除**

**封闭库保护规则**（2026-06-08 修复）：  
兜底逻辑原来会绕过封闭库过滤，已修复为兜底时依然保持 domain 过滤，只放宽得分阈值。

### 4.3 is_index 库索引聚合机制

当检索命中带有 `is_index=True` 标记的 chunk 时（即库清单文件），系统自动拉取该文件的**全部 chunk** 并置于上下文最前面，优先供 LLM 参考，用于准确回答"有多少条"/"有哪些"类问题。

触发条件：MD 文件所在目录下有以"案例"结尾的子目录（如 `客关部/排雷库/` 目录下有 `排雷库案例/`）。

## 五、回答精准度改进记录

### 5.1 数量类问题答错（如"有多少案例"）

**问题**：问"客关部排雷库有多少案例"，回答说 5 条（实为 123 条）。  
**根因**：LLM 数了参考资料里出现的案例卡片片段数量，没有读到排雷库清单文件的总数。  
**修复**：
1. 建立库索引清单文件（`排雷库清单.md`、`设计部案例清单.md`）并标记 `is_index=True`
2. 命中时整份聚合到上下文最前面
3. 在 user_prompt 中加入 `index_hint`，指示 LLM 从索引文件统计，不要自行数片段

**`index_hint` 演进**：
- 第一版：硬编码"排雷库清单总览"和"123条"字样 → 在设计部查询时误导 LLM → 回答"未找到"
- 第二版（当前）：通用描述"含总览/清单/介绍的索引文件，从明细列表中统计" → 适用所有库

### 5.2 封闭库内容泄漏

**问题**：问"泳池相关注意事项"（未指定部门）→ 返回了客关部排雷库内容。  
**根因**：兜底逻辑（当过滤后无结果时）完全跳过了 `_passes_filter`，封闭库绕过生效。  
**修复**（2026-06-08）：兜底改为仅放宽得分阈值，`_passes_filter`（含封闭库检查）继续生效。

### 5.3 多版本重复图片

**问题**：用心库 V1 和 V2 有相同条目（A-1 龙头），LLM 各引一张图，输出两张几乎一样的图。  
**根因**：V2 是 V1 升级版，大量内容原样保留，两个文件都进了库，检索都命中。  
**修复**：图片引用指令加入"若多版本图片内容相似，只保留最新版的一张"。

### 5.4 PPTX 识别质量差

**问题**：设计部惊喜库 V1/V3、用心库 V1 原为 PPTX，python-pptx 对分组图形/SmartArt 无法识别，入库效果极差。  
**修复**：用 LibreOffice headless 将 PPTX 转为 PDF，再用 PyMuPDF 提取文字 + 截图，重新入库。

## 八、关键命令速查

```bash
# 启动飞书机器人
bash start_feishu.sh

# 重建 BM25 索引（入库后必须执行）
python3 -c "
import chromadb, pickle, os
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
r = col.get(include=['documents','metadatas'])
corpus = list(zip(r['documents'], r['metadatas']))
with open('bm25_index.pkl.tmp','wb') as f: pickle.dump(corpus,f)
os.replace('bm25_index.pkl.tmp','bm25_index.pkl')
print(len(corpus))
"

# 强制重入某个 MD 文件
python3 -c "
import chromadb
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
r = col.get(where={'source': '设计部/20260608_设计部案例清单.md'})
col.delete(ids=r['ids'])
print(len(r['ids']))
"
venv/bin/python index_reviewed_md.py

# 设计部案例重制（含截图）
venv/bin/python reindex_design_cases.py

# 查询某 domain 的 chunk 数
python3 -c "
import chromadb
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
r = col.get(where={'domain':'客关部'})
print(len(r['ids']))
"
```
