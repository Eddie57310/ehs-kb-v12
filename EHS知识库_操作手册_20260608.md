## 一、环境准备

### 1.1 Python 虚拟环境

所有脚本须在 v11 专属 venv 中运行：

```bash
cd ~/doc_parser_v11

# 方式一：激活（每次开新终端都要做）
source venv/bin/activate

# 方式二：直接用绝对路径（推荐，无需激活）
venv/bin/python <脚本名>.py
```

### 1.2 环境变量（.env）

根目录下须有 `.env` 文件：

```
FEISHU_APP_ID=cli_xxxx
FEISHU_APP_SECRET=xxxxxxxx
VOLCODING_API_KEY=ark-xxxx
```

飞书 WS 服务启动时自动读取。此文件不入 git（已加入 `.gitignore`）。

### 1.3 MinerU 配置

配置文件：`~/magic-pdf.json`

```json
{
  "device-mode": "cpu"
}
```

无 GPU 环境下 `"device-mode"` 必须设为 `"cpu"`，否则 MinerU 启动报错。

## 二、数据入库流程

### 2.1 入库路线选择

| 路线 | 适用文件类型 | 有无 MD 中间文件 |
|------|-------------|----------------|
| 常规路线 | 公司内部制度、国家规定、工管部文件 | 有，可人工校核 |
| EHS案例路线（Vision LLM） | EHS案例 PDF（负面/正面/警示） | 无，终端日志审核 |
| 设计部案例路线（截图入库） | 设计部案例 PDF（PPTX 须先转 PDF） | 无 |

---

### 2.2 常规路线（公司内部 / 国家规定 / 工管部）

**步骤一：将文件放入 Local_KB**

```
Local_KB/
├── 公司内部/      ← 公司内部制度、管理手册
├── 国家规定/      ← 国家法规、行业标准、地方规范
└── 工管部/        ← 工管部专属文件
    └── <文件名>.pdf / .docx
```

**步骤二：MinerU 解析，生成 MD 文件**

```bash
venv/bin/python rebuild_md.py
```

- 扫描 Local_KB，对尚未解析的 PDF/Word 执行 MinerU 解析
- 输出到 `reviewed_md/<对应路径>.md`
- 已解析过的文件自动跳过，只处理新增文件
- 耗时视文件大小，通常 1～5 分钟/文件

**步骤三：人工审核 MD 文件（必做，不可省略）**

启动 MD 审核工具：

```bash
venv/bin/python md_reviewer.py
# 浏览器打开 http://localhost:8084
```

**审核必做事项**：

| 检查项 | 操作 |
|--------|------|
| 目录索引页 | 文档开头的目录（章节名+页码列表）**直接删掉**，对检索有害 |
| 合并单元格 | 手动展开：将合并项在每行中重复填写 |
| 表格前后空行 | 确保表格前后各有一个空行，否则 Markdown 不渲染为表格 |
| 图片插入 | 需要图片时，插入标记 `[📷 /绝对路径/图片.png]` |

**表格修复工具**：浏览器直接打开 `~/doc_parser_v11/table_fixer.html`，粘贴残缺表格，自动修复并预览。

**步骤四：入库**

```bash
venv/bin/python index_reviewed_md.py
```

- 读取 `reviewed_md/` 下的 MD 文件
- 按 `##` 二级标题切块，生成 chunk（每块含 `chunk_seq` 序号和父标题前缀）
- Embedding（bge-m3）→ 写入 ChromaDB
- 已入库的 source 路径自动跳过，重跑安全

**步骤五：重建 BM25 索引（入库后必须执行）**

```bash
python3 -c "
import chromadb, pickle, os
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
r = col.get(include=['documents','metadatas'])
corpus = list(zip(r['documents'], r['metadatas']))
with open('bm25_index.pkl.tmp','wb') as f: pickle.dump(corpus,f)
os.replace('bm25_index.pkl.tmp','bm25_index.pkl')
print(f'BM25 重建完成，共 {len(corpus)} 条')
"
```

```bash
# 2. 执行入库
```

### 2.4 设计部案例路线（截图入库）

**PPTX 文件须先转 PDF**：

```bash
libreoffice --headless --convert-to pdf \
  --outdir Local_KB/设计部/案例/ \
  <文件.pptx>
```

**执行入库**：

```bash
venv/bin/python reindex_design_cases.py
```

脚本自动执行：
1. 备份 `chroma_db`
2. 删除该批文件的旧 chunk
3. PyMuPDF 提取文字 + 整页截图（Matrix 2.0，约 144 DPI）
4. 入库 ChromaDB
5. 重建 BM25

---

### 2.5 强制重入单个文件

当某文件已入库但需要更新时，`index_reviewed_md.py` 默认会跳过它，须先手动删除旧 chunk：

```bash
# 步骤1：删除旧 chunk（替换 source 路径为实际值）
python3 -c "
import chromadb
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
source = '设计部/20260608_设计部案例清单.md'   # ← 修改为实际路径
r = col.get(where={'source': source})
print(f'将删除 {len(r[\"ids\"])} 个 chunk')
col.delete(ids=r['ids'])
print('删除完成')
"

# 步骤2：重新入库
venv/bin/python index_reviewed_md.py

# 步骤3：重建 BM25（见 2.2 步骤五）
```

## 三、新增案例库索引文件

每个案例库须配套一份索引清单 MD，用于准确回答"有多少条"/"有哪些"类问题。

### 3.1 目录结构准备

```
Local_KB/<部门>/<库名>/
├── YYYYMMDD_<库名>清单.md   ← 索引清单（须有且仅有一个）
└── <库名>案例/              ← 案例原件（目录名须以"案例"结尾）
    ├── xxx.pdf
    └── ...
```

"以'案例'结尾的子目录"的存在会触发系统自动将清单 MD 标记为 `is_index=True`。

### 3.2 清单文件内容模板

```markdown
# [库名]介绍
（1-2句话说明库的定位、来源、保密级别）

# [库名]总览
共收录 XX 条 [案例/条目]。

| 分类 | 数量 |
|------|------|
| A类  | XX条 |
| B类  | XX条 |

# A类 分类明细

| 编号 | 案例名称 |
|------|---------|
| A-001 | 名称一  |
| A-002 | 名称二  |

# B类 分类明细
...
```

**关键要求**：
- 总览表必须写明总条数（LLM 的权威数字来源）
- 各维度明细列表必须完整
- 标题使用 `#`/`##` 层级
- 文件名含日期前缀（`YYYYMMDD_`）

### 3.3 入库步骤

```bash
# 1. 复制到 reviewed_md（或直接在 reviewed_md 下创建）
cp Local_KB/<部门>/<库名>/xxx清单.md \
   reviewed_md/<部门>/<库名>/xxx清单.md

# 2. 入库
venv/bin/python index_reviewed_md.py

# 3. 重建 BM25
# （见 2.2 步骤五）
```

### 3.4 验证

```bash
# 验证 is_index 标记是否生效
python3 -c "
import chromadb
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
r = col.get(where={'is_index': True})
for s in sorted(set(m['source'] for m in r['metadatas'])):
    print(s)
"
# 应能看到新增的清单文件 source
```

然后在飞书问"[部门] [库名]有多少案例"，确认数字与清单总览一致。

### 6.2 查看知识库各 domain 分布

```bash
python3 -c "
import chromadb
from collections import Counter
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
r = col.get(include=['metadatas'])
domains = Counter(m.get('domain','未知') for m in r['metadatas'])
for d, cnt in sorted(domains.items(), key=lambda x: -x[1]):
    print(f'{d:12s}: {cnt} chunks')
print(f'总计: {sum(domains.values())} chunks')
"
```

### 6.3 查看所有 is_index 文件

```bash
python3 -c "
import chromadb
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
r = col.get(where={'is_index': True})
for s in sorted(set(m['source'] for m in r['metadatas'])):
    print(s)
"
```

### 6.4 查看某文件的 chunk 详情

```bash
python3 -c "
import chromadb
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
source = '设计部/20260608_设计部案例清单.md'   # ← 修改为实际路径

```

### 6.5 查找 metadata 为 None 的孤立 chunk

```bash
python3 -c "
import chromadb
c = chromadb.PersistentClient(path='chroma_db')
col = c.get_collection('langchain')
r = col.get(include=['metadatas'])
orphans = [r['ids'][i] for i, m in enumerate(r['metadatas']) if not m]

if orphans:
    print('IDs:', orphans[:5])
"
```

## 七、跨版本数据迁移

ChromaDB 支持按 metadata 过滤导出，可将某个 domain 从一个版本迁移到另一个版本。

**前提**：两个版本的 Embedding 模型必须一致（V10/V11 均为 `BAAI/bge-m3`，向量可直接搬，无需重新编码）。

```python
import chromadb

v11 = chromadb.PersistentClient(path='/path/to/v11/chroma_db')
v10 = chromadb.PersistentClient(path='/path/to/v10/chroma_db')
v11_col = v11.get_collection('langchain')
v10_col = v10.get_collection('langchain')

# 1. 从源库按 domain 提取
results = v11_col.get(
    where={"domain": "客关部"},
    include=["documents", "metadatas", "embeddings"]
)

# 2. 迁移前检查目标库是否已有同源数据
for source in set(m['source'] for m in results['metadatas']):
    existing = v10_col.get(where={'source': source})
    if existing['ids']:
        print(f'目标库已有 {len(existing["ids"])} 条同源数据，先删除')
        v10_col.delete(ids=existing['ids'])

# 3. 写入目标库（upsert 防冲突）
v10_col.upsert(
    ids=results["ids"],
    documents=results["documents"],
    metadatas=results["metadatas"],
    embeddings=results["embeddings"]
)
```

迁移后须重建目标版本的 BM25 索引（见 2.2 步骤五）。

**注意**：如果 chunk 中有 `[📷 路径]` 标记，路径可能因版本目录不同需要调整。

## 八、常见问题处理

### 文件已入库，修改后重跑 index_reviewed_md.py 不生效

**原因**：增量逻辑检测到 source 已存在，自动跳过。  
**处理**：先删除旧 chunk，再重跑。见 2.5 节。

---

### 飞书机器人启动等待时使用 sleep 被阻断（Exit code 144）

**原因**：`sleep` 命令在某些环境下被系统拦截。  
**处理**：改用条件等待：

```bash
until grep -q "connected to wss" feishu_run.log; do sleep 3; done
```

---

### BM25 检索没有更新

**原因**：入库后未重建 `bm25_index.pkl`。  
**处理**：执行 2.2 步骤五的重建命令。

---

### Chunk 审核工具（8085）页面崩溃

**原因**：ChromaDB 中有 metadata 为 None 的孤立 chunk，旧版代码未做 None-safe 处理。  
**处理**：`chunk_reviewer.py` 已内置 None-safe 保护，重启服务即可。查找孤立 chunk 见 6.5 节。

---

### MinerU 解析报缺失依赖

**已装入 v11 venv 的依赖**：cv2、ultralytics、doclayout-yolo、rapid-table、unimernet、shapely、pyclipper、opencv-python-headless。

如遇新的缺失报错：

```bash
venv/bin/pip install <缺失包名>
```

---

### Chunk 审核工具修改后未生效（视图缓存）

**原因**：`chunk_reviewer` 在内存中缓存了 ChromaDB 视图，修改 ChromaDB 后不自动刷新。  
**处理**：重启 `chunk_reviewer`，它会重新加载最新状态。
