# EHS 智慧助手知识库 — 方案说明

**版本**：V11  
**整理日期**：2026-06-08  
**项目路径**：`~/doc_parser_v11/`

---

## 一、系统概述

### 1.1 这是什么

一个**会说话的 EHS 知识库**，以飞书机器人为交互入口。

把公司内部制度、国家法规、EHS 案例全部录入系统后，员工可以直接在飞书上用自然语言提问，系统自动找到相关内容并整合成清晰的回答。

> 举例：提问"高处作业安全要求是什么？"→ 系统自动检索相关规定，综合多份文件，给出结构化的完整回答。

### 1.2 解决了什么问题

| 以前 | 现在 |
|---|---|
| 要找一条规定，得翻几十份 PDF | 直接问，几秒出答案 |
| 文件越来越多，没人记得住 | 知识库统一管理，随时可查 |
| 新员工不知道去哪找制度 | 直接在飞书提问即可 |
| 案例图文分散，难以利用 | 图片与文字一同召回，原图可见 |

## 二、系统架构

### 2.1 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| 向量数据库 | ChromaDB | 持久化存储，目录 `chroma_db/` |
| Embedding 模型 | BAAI/bge-m3 | 本地 CPU 推理，多语言支持 |
| BM25 关键词索引 | 自建 | 序列化为 `bm25_index.pkl`，jieba 分词 |
| Reranker | bge-reranker-v2-m3 | 本地 CPU，精排 top_k=8 |
| LLM（问答） | 豆包 doubao-seed-2.0-pro | 云端 API，只接收检索结果 |
| 机器人接入 | 飞书 WebSocket 长连接 | `feishu_ws_server_v11.py` |
| PDF 解析 | MinerU（CPU 模式）+ PyMuPDF | 常规文档路线 |
| PPTX 解析 | LibreOffice headless → PyMuPDF | PPTX 转 PDF 后处理 |

**私有化部署说明**：Embedding、Reranker、BM25 均在公司服务器本地运行。LLM 调用豆包云端 API，但只发送用户问题和检索到的参考文本片段，知识库原文不上传外部。

### 2.2 整体检索流程

```
员工提问（飞书）
   ↓
[1] 双轨检索：
    ChromaDB 向量检索（k=20，bge-m3 embedding）
  + BM25 关键词检索（k=20，jieba 分词）
   ↓
[2] RRF 互惠排名融合（Reciprocal Rank Fusion）
   ↓
[3] Domain / 封闭库过滤（见第三章）
   ↓
[4] is_index 索引聚合（命中清单文件时整份拉取，见第四章）
   ↓
[5] Reranker 精排（bge-reranker-v2-m3，top_k=8）
   ↓
[6] LLM 生成答案（豆包 doubao-seed-2.0-pro）
   ↓
输出答案（含来源文件、生效时间）
```

**设计选型说明**：
- **双轨检索**：语义向量检索擅长近义词、概念匹配；BM25 关键词检索擅长精确词命中；RRF 融合兼顾两者，互补。
- **Reranker 精排**：从 20 个 RRF 候选中精选 8 条，bge-reranker-v2-m3 在中文理解上显著优于仅依赖 embedding 距离的排序。
- **top_k=8、Reranker 关 fp16**：在无 GPU 的 CPU 环境下，fp16 模式不稳定，关闭后延迟可接受。

## 三、知识库结构与封闭库机制

### 3.1 知识库分类

#### 公开库（通用问答均可检索）

| Domain | 路径 | 内容说明 |
|--------|------|---------|
| EHS案例 | `Local_KB/EHS案例/` | 负面事故案例、正面标准化案例、警示手册 |
| 公司内部 | `Local_KB/公司内部/` | 华润置地内部制度、项目部规定、管理手册 |
| 国家规定 | `Local_KB/国家规定/` | 国家法律法规、行业标准、浙江地方规范等 |

#### 封闭库（必须在提问中写出完整部门名才会检索）

| Domain | 路径 | 内容说明 |
|--------|------|---------|
| 客关部 | `Local_KB/客关部/` | 排雷库（123条）、物业检查标准、维保指引、客户风险检查表 |
| 设计部 | `Local_KB/设计部/` | 惊喜库V1/V2/V3（134条卡片）、用心库V1/V2、沧海桑田计划2.0 |
| 工管部 | `Local_KB/工管部/` | 施工工艺标准、示范区提质增效案例集 |
| 合约部 | `Local_KB/合约部/` | 杭州公司缺陷案例库（住宅篇） |

封闭库的内容属于特定部门专属，不在通用提问时暴露，避免信息越权访问。

### 3.2 Domain 路由规则

系统在解析提问时，按以下规则确定检索范围：

| 提问关键词 | 检索范围 |
|-----------|---------|
| `EHS案例` / `公司内部` / `国家规定` | 仅在该 domain 内检索 |
| `客关部` / `设计部` / `工管部` / `合约部` | 仅在该封闭库内检索 |
| 无以上关键词 | 仅在三大公开库中检索，**封闭库自动排除** |

**重要**：封闭库触发词须写**完整部门名**。"客关部"才能触发，"客关"不会触发。

### 3.3 封闭库保护机制（兜底逻辑设计）

当过滤后无结果时，系统会执行"兜底"逻辑放宽得分阈值，扩大候选集。兜底逻辑的设计原则：**只放宽得分阈值，不放宽 domain 过滤**。

```python
# 兜底：放宽得分阈值，但仍保持封闭域过滤
all_filtered = [(doc, score) for doc, score in docs_and_scores if _passes_filter(doc)]
valid_with_scores = _rerank(question, all_filtered, top_k=top_k)
```

封闭库在兜底时同样不会泄漏给无权限的提问（2026-06-08 修复了兜底时绕过过滤的 bug）。

## 四、案例库索引聚合机制（is_index 方案）

### 4.1 背景问题

知识库里的案例类文件（如排雷库、惊喜库）在入库时按页或按段落切块，每个 chunk 只包含一条案例的局部内容。当用户问"有多少案例"或"有哪些案例"时：

1. 向量检索命中若干案例卡片片段（通常 5～8 条）
2. LLM 数了上下文里出现的案例片段，回答说"共 5 条"
3. 实际上库里有 123 条，答案严重错误

**根本原因**：向量检索是"相似度匹配"，只返回最相关的若干片段，不返回全部内容；LLM 没有"扫描全库计数"的能力，只能基于上下文回答。

### 4.2 解决方案

为每个案例库**人工维护一份"索引清单 MD 文件"**，把案例的总数、分类、明细列表写清楚，入库时自动标记为索引文件（`is_index=True`），检索命中时整份带入上下文，由 LLM 直接读数。

### 4.3 整体流程

```
用户提问（"有多少案例" / "有哪些案例"）
        ↓
混合检索（向量 + BM25）
        ↓
命中某个 is_index 文件的任意 chunk
        ↓
【索引聚合】：从 ChromaDB 拉出该文件全部 chunk，按 chunk_seq 排序
        ↓
整份索引置于上下文最前面
        ↓
LLM 收到 index_hint 指令："从索引文件的明细列表中统计"
        ↓
LLM 直接读索引，给出准确数字
```

### 4.4 三个关键组件

| 组件 | 位置 | 作用 |
|------|------|------|
| `is_index` 标记 | ChromaDB chunk metadata | 标识该 chunk 来自索引文件 |
| 索引聚合逻辑 | `feishu_ws_server_v11.py` 约 982 行 | 命中即从 ChromaDB 整份拉取 |
| `index_hint` 指令 | user_prompt（问题后面） | 告诉 LLM 从索引统计，不要数片段 |

### 4.5 is_index 自动识别规则

系统在入库时自动识别索引文件（`index_reviewed_md.py`）：

> **规则**：一个 MD 文件，若其所在目录（在 `Local_KB/` 中）下存在名称以"**案例**"结尾的子目录，则该文件标记为 `is_index=True`。

```
Local_KB/
├── 客关部/
│   └── 排雷库/
│       ├── 20260401_排雷库清单.md   ← ✅ is_index=True
│       │                              （同级有"排雷库案例"子目录）
│       └── 排雷库案例/
│           └── xxx.pdf
│
└── 设计部/
    ├── 20260608_设计部案例清单.md   ← ✅ is_index=True
    │                                  （同级有"案例"子目录）
    └── 案例/
        └── 惊喜库v1.pdf
```

每个 chunk 带有 `chunk_seq`（从 1 开始的序号），聚合时按此还原文件顺序。

### 4.6 index_hint 的设计考量

`index_hint` 放在"问题后面、格式要求前面"的指令区域，而不是参考资料区域。

**原因**：LLM 对 prompt 不同区域的权重不同。放在参考资料区域会被当作普通文本，权重低；放在指令区域（问题后面）权重最高，LLM 会优先遵守。

**演进历史**：
- v1（已废弃）：硬编码"排雷库清单总览"和"共收录123条问题"字样 → 设计部查询时 LLM 找不到匹配的字样，回答"未在知识库中找到"
- v2（当前）：通用描述，不提具体库名或数字，适用所有库

### 4.7 现有索引文件

| 文件 | 路径 | 覆盖内容 |
|------|------|---------|
| 排雷库清单 | `客关部/排雷库/20260401_排雷库清单.md` | 123条问题，按风险类型、专业、阶段分类 |
| 设计部案例清单 | `设计部/20260608_设计部案例清单.md` | 6个文件总览（134条惊喜库卡片+用心库+沧海桑田） |

### 4.8 局限性

| 问题 | 说明 |
|------|------|
| 人工维护负担 | 案例更新时须同步修改清单的数字和明细列表；漏改则 LLM 给出过期数字 |
| 识别规则边界 | 同一目录下若有多个 MD 文件，每个都会被标记；实践中每库只放一个清单 MD 可避免 |
| token 占用 | 整份索引（如 18 个 chunk ≈ 5400 字）置于上下文，索引过大时可能挤占其他结果 |
| 仅适用于 MD 索引 | 设计部案例 PDF 直接入库，用心库/沧海桑田的详细条目数需在 MD 清单中手动维护 |

## 五、数据入库路线设计

系统支持三条入库路线，根据文件类型选择：

### 5.1 常规路线（PDF/Word → MinerU → MD → 人工审核 → 入库）

适用：公司内部制度、国家规定、工管部文件

```
PDF/Word 原文件
    ↓ rebuild_md.py（MinerU 解析）
reviewed_md/<路径>.md（Markdown 中间文件）
    ↓ 人工审核（必做）
    ↓ index_reviewed_md.py（切块 + Embedding）
ChromaDB（向量库）
```

**此路线有 MD 中间文件**，允许人工校核、插入图片、修复表格，是质量最可控的路线。

### 5.2 EHS 案例路线（Vision LLM 直接入库）

适用：EHS 案例 PDF（负面/正面/警示手册）

```
PDF 原文件（Local_KB/EHS案例/）
    ↓ sync_kb_v11.py --domain EHS案例
    ↓ 逐页 Vision LLM 识别
ChromaDB（无中间文件）
```

**此路线无 MD 中间文件**，通过终端日志实时审核；识别质量依赖 Vision LLM，图文混排效果好。

### 5.3 设计部案例路线（截图 + 文字提取入库）

适用：设计部案例 PDF（PPTX 须先转 PDF）

```
PDF 原文件（Local_KB/设计部/案例/）
    ↓ reindex_design_cases.py
    ↓ PyMuPDF 提取文字 + 整页截图（Matrix 2.0，约 144 DPI）
ChromaDB（无中间文件，chunk 含截图路径标记）
```

**PPTX 说明**：python-pptx 对分组图形/SmartArt 无法识别，须先用 LibreOffice headless 转为 PDF 再入库。

### 5.4 入库公共规则

- `index_reviewed_md.py` 有增量逻辑：已入库的 source 路径自动跳过，重跑安全
- 每次入库后须重建 `bm25_index.pkl`，否则关键词检索不更新
- 每次大批量操作前须备份 `chroma_db/`

## 六、源文件编制要求

本章规定向知识库上传的源文件编制标准，确保文件能被系统准确解析和入库。

### 6.1 表格规范

#### 禁止使用合并单元格

| 不符合要求 | 符合要求 |
|-----------|---------|
| 跨行/跨列合并单元格（产生空白格） | 每行独立，合并项重复填写 |

**示例**：原 Excel 中"高风险"合并了两行，应展开为：

| 风险等级 | 单位名称 |
|---|---|
| 高风险 | 总承包单位 |
| 高风险 | 幕墙单位 |

> **原因**：知识库解析系统不支持合并单元格，合并将导致数据错位或丢失，影响检索准确性。

#### 禁止以图片形式呈现表格

须使用 Word / Excel 原生表格功能制作，文字可复制选中。

> **原因**：图片形式的表格无法被系统直接提取文字，OCR 识别准确率低，导致入库数据残缺。

### 6.2 图片规范（针对排雷库类文件）

**图片须包含问题描述**

飞书聊天不支持"文本+图片+文本"富文本交替排版，系统先输出全部文字，再统一发送图片。用户看到图片时，上下文可能已经滚过，无法直接对应具体问题。

要求（任选其一）：
- 在图片上直接标注简要问题描述（加文字水印/标题栏）
- 在源文件中将问题描述文字紧贴图片上方，截图时一起截入

**目标**：单独看一张图片，就能看出它对应的是什么问题。

### 6.3 案例库目录结构规范

每个"库"作为一个独立文件夹，库根目录只放索引清单，案例原件放入以"案例"结尾的子文件夹：

```
客关部/
└── 排雷库/                       ← 库文件夹
    ├── 20260401_排雷库清单.md     ← 索引清单（库根，唯一）
    └── 排雷库案例/                ← 案例原件（须以"案例"结尾）
        ├── 排雷库V1.0.pdf
        ├── 排雷库V2.0.pdf
        └── 排雷库V3.0.pdf
```

**要求**：
- 库根目录**有且仅有一个**索引清单 `.md`
- 案例原件全部放进名为"<库名>案例"的子文件夹（目录名须以"案例"二字结尾）

> **原因**：系统据此自动识别——库根清单 .md 被标记为索引并在检索时整份返回；"*案例"子目录内的文件不会被当作清单重复入库。

### 6.4 索引清单文件编制规范

索引清单集中承载：库的介绍说明 + 完整条目清单（各维度/分项、编号、名称、条数）。案例卡片中**不再重复**清单与说明内容。

**结构模板**：

```markdown
# [库名]介绍
（1-2句话说明库的定位、来源、保密级别）

# [库名]总览
共收录 XX 条 [案例/条目]。

| 分类 | 数量 |
|------|------|
| A类  | XX条 |
| B类  | XX条 |

# A类 分类明细

| 编号 | 案例名称 |
|------|---------|
| A-001 | 名称一  |
| A-002 | 名称二  |
```

**关键要求**：

1. **总览表必须写明总条数**（如"共收录 123 条"），是 LLM 的权威数字来源
2. **各维度的明细列表必须完整**，LLM 可从中逐条数
3. 标题使用 `#`/`##`/`###` 层级组织，切块时按层级分割
4. 文件名含日期前缀（`YYYYMMDD_`）
5. "共多少条 / 各维度条数"**只在清单里写一处**，不要在多个文件里各写一份（避免多版本数字冲突）

### 6.5 排雷库案例卡片字段规范

每张排雷卡作为一个独立案例编制，字段齐全、格式统一：

```
编号：GQ-DX010
案例名称：……
排雷节点：……
问题分类：……
风险描述：……
风险原因：……
预防建议：……
涉及专业：……
涉及项目：……
产生后果：……
（问题描述文字 + 截图，紧随其后）
```

**要求**：
- 字段名后统一用中文冒号"："（"排雷节点："而非"排雷节点  "）
- 编号严格按"维度-分项+数字"格式（如 `GQ-DX010`），**勿多写 0**（`GQ-DX0010` 会与清单对不上号）
- 一张卡 = 一个案例，不混入其他案例、清单页或目录

### 6.6 一般文档编排建议

- 在文字能描述清楚的情况下，尽量不制作表格（OCR 识别困难）
- 页码编排须规范，推荐层级：一、1、（1）、1）

## 七、回答精准度改进记录

记录系统在迭代过程中发现并修复的检索/回答质量问题，供后续维护参考。

### 7.1 数量类问题答错

**现象**：问"有多少案例"，回答说 5 条（实为 123 条）。  
**根因**：LLM 数了参考资料里出现的案例片段数量，没有读到总览数字。  
**修复**：建立 is_index 库索引方案（详见第四章）。

### 7.2 index_hint 误导（v1 缺陷）

**现象**：问"设计部惊喜库有多少案例"，回答"未在知识库中找到"。  
**根因**：index_hint 第一版硬编码"排雷库清单总览"和"共收录123条"字样，LLM 在设计部上下文中找不到对应字样，误判为无结果。  
**修复**：index_hint 改为通用描述，不提具体库名或数字。

### 7.3 封闭库内容泄漏

**现象**：问"泳池相关注意事项"（未指定部门）→ 返回了客关部排雷库内容。  
**根因**：兜底逻辑完全跳过了 `_passes_filter`，封闭库过滤被绕过。  
**修复**（2026-06-08）：兜底改为仅放宽得分阈值，`_passes_filter`（含封闭库检查）继续生效。

### 7.4 多版本重复图片

**现象**：用心库 V1 和 V2 有相同的 A-1 条目，LLM 各引一张图，输出两张几乎一样的图。  
**根因**：V2 是 V1 升级版，大量内容原样保留，两个文件都进了库，检索都命中。  
**修复**：图片引用指令加入"若多版本图片内容相似，只保留最新版的一张"。

### 7.5 PPTX 识别质量差

**现象**：设计部 PPTX 文件入库后检索效果极差，图文内容大量缺失。  
**根因**：python-pptx 对分组图形/SmartArt 无法识别。  
**修复**：改用 LibreOffice headless 将 PPTX 转为 PDF，再用 PyMuPDF 提取文字 + 截图，重新入库。

## 八、用户使用说明

在飞书直接 @ 机器人提问，无需培训。

### 8.1 常用技巧

| 用法 | 说明 |
|------|------|
| 直接提问 | 自动在三大公开库（EHS案例/公司内部/国家规定）中检索 |
| 加"精准匹配" | 提高答案相关性，倾向于关键词匹配 |
| 加"开启分析" | 回答末尾附带引用来源清单 |
| 加"EHS案例"/"公司内部"/"国家规定" | 只在该公开库内检索 |
| 加"客关部"/"设计部"/"工管部"/"合约部" | 检索对应封闭库（须写完整部门名） |
| 发送 `/目录` | 查看知识库文件列表 |
| 发送 `/说明` | 查看系统功能说明和封闭库使用指南 |

### 8.2 封闭库使用注意

- 封闭库须写**完整部门名**：写"客关部"才能触发，写"客关"不会触发
- 示例：
  - ✅ "客关部排雷库中有哪些泳池相关问题"
  - ❌ "客关的排雷库里有什么泳池问题"（不会检索客关部）
  - ✅ "设计部惊喜库 V3 有哪些景观案例"
  - ❌ "设计的案例库里面有没有景观相关的"
