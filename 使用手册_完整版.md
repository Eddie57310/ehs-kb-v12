## 0. 进入环境（每次操作前先做这一步）

```bash
cd ~/doc_parser_v11
source venv/bin/activate
```

激活成功后命令行前会出现 `(venv)`。退出环境用 `deactivate`。

**不想激活也行**：所有命令把 `python` 换成 `venv/bin/python` 前缀即可，效果一样。例如：
```bash
venv/bin/python rebuild_md.py
```

> 本手册后面的命令默认你已经 `cd ~/doc_parser_v11`。没激活 venv 的话，把 `python` 读成 `venv/bin/python`。

## 1. 系统总览

### 1.1 知识库目录结构

源文件统一放在 `Local_KB/<专业部门>/...`，顶级文件夹名就是 **domain（专业域）**：

```
Local_KB/
├── 公司内部/      # 常规文档（走 MinerU/MD 路线）
├── 国家规定/      # 常规文档
├── 工管部/        # 常规文档 ← 你新增的
├── 客关部/        # 含「排雷库」等案例库
├── 设计部/
└── EHS案例/       # 特殊：走 Vision LLM 路线，跳过 MD
```

### 1.2 两条入库路线（关键！别走错）

| 路线 | 适用 domain | 入口脚本 | 是否生成 MD | 审核工具 |
|------|------------|---------|------------|---------|
| **常规文档** | 公司内部 / 国家规定 / 工管部 / 设计部 等 | `rebuild_md.py` → `index_reviewed_md.py` | ✅ 生成 `reviewed_md/` | md_reviewer (8084) |
| **EHS案例（Vision）** | `EHS案例`，以及任意部门下名为「案例」的子目录 | `sync_kb_v11.py --domain EHS案例` | ❌ 直接入库 | chunk_reviewer (8085) |

> **工管部是常规文档**，所以走第一条线。`sync_kb_v11.py` 是 EHS案例专用，工管部千万别用它。

### 1.3 端口表

| 端口 | 服务 | 启动命令 | 用途 |
|------|------|---------|------|
| 8080 | api_server_v11.py | `python api_server_v11.py` | 后端检索网关（FastAPI）|
| 8081 | web_ui.py | `python web_ui.py` | 网页版聊天 UI |
| 8084 | md_reviewer.py | `python md_reviewer.py` | **MD 文件人工审核**（常规文档）|
| 8085 | chunk_reviewer.py | `python chunk_reviewer.py` | **Chunk 数据质检**（EHS案例/已入库块）|

访问：浏览器开 `http://<本机IP>:端口`（本机就是 `http://127.0.0.1:端口`）。

### 1.4 技术栈

- **向量库**：ChromaDB（`chroma_db/`），embedding 模型 `BAAI/bge-m3`（CPU）
- **关键词检索**：BM25（`bm25_index.pkl`），与向量做 RRF 混合 + reranker 重排
- **PDF 解析**：MinerU（`magic-pdf`，CPU 模式）
- **格式转换**：LibreOffice（`.doc/.ods → .docx/.xlsx/.pdf`）
- **问答大模型**：飞书链路默认云端豆包 `doubao-seed-2.0-pro`（不是本地 qwen）

## 2. 常规文档入库流程（工管部走这条）

**四步，第 2 步和第 4 步之间必须人工审核，不能自动连跑。**

### 步骤 1：放源文件
把文件放进 `Local_KB/<部门>/`，比如 `Local_KB/工管部/`。支持的格式：

| 格式 | 是否支持 | 说明 |
|------|---------|------|
| `.pdf` | ✅ | MinerU 解析，保留表格/公式为文本 |
| `.docx` | ✅ | LibreOffice→HTML→MD，序号/页眉/表格完整保留 |
| `.xlsx` / `.xls` / `.ods` | ✅ | openpyxl 转 MD（`.ods` 先转 `.xlsx`）|
| **`.doc`（老格式）** | ❌ **不支持** | 必须先转成 `.docx`（见第 6.1 节）|

### 步骤 2：解析成 Markdown
```bash
python rebuild_md.py
```
- 扫描整个 `Local_KB/`，输出到 `reviewed_md/`
- **增量逻辑**：`reviewed_md/` 里已存在的 MD 自动 skip，所以只处理新文件，重跑安全
- 自动跳过 `EHS案例` 和名为「案例」的子目录
- 日志在 `logs/`

### 步骤 3：人工审核 MD（端口 8084）
```bash
python md_reviewer.py
```
浏览器开 `http://127.0.0.1:8084`，三列界面：**源文件 PDF 全页 | 编辑区 | 实时预览**。

审核要点（**直接影响检索质量**）：
- **删掉文档开头的目录索引**（章节名+页码那种），对检索有害
- **修复表格**：Markdown 不支持合并单元格，跨行/跨列的要手动拆开重复填写；表格**前后必须留空行**否则不渲染
  - 工具：浏览器打开 `table_fixer.html`，粘贴残缺表格自动修复+预览
- **插入图片**：界面支持从源 PDF 框选截图插入
- 保存历史记录在 `reviewed_md/.save_history.json`

### 步骤 4：切片入库
```bash
python index_reviewed_md.py            # 增量：只入未入库的文件
python index_reviewed_md.py --force    # 全量重建（先清空旧向量，慎用）
python index_reviewed_md.py --file 工管部/xxx/foo.md   # 只入单个文件
```
- 切片参数：CHUNK_SIZE=2000，OVERLAP=200
- 增量逻辑：已入库文件自动 skip，重跑安全
- 库根索引 `.md` 会自动标 `is_index`，检索命中后按 `chunk_seq` 整份聚合

### 步骤 5：重建 BM25 + 让机器人生效
入库只更新了向量库，**BM25 索引和在线服务还是旧的**。重启飞书机器人即可自动重建 BM25：
```bash
bash start_feishu.sh
```
> 飞书服务启动时会后台自动从 `chroma_db` 重建 `bm25_index.pkl`，并监控该文件热加载。

## 3. EHS案例入库流程（Vision LLM 路线）

仅用于 `EHS案例` domain 或部门下「案例」子目录。

```bash
python sync_kb_v11.py                      # 处理全部 domain
python sync_kb_v11.py --domain EHS案例     # 只处理指定顶级目录
```
- 逐页判断：文字少于阈值（<100 字）的页走 Vision LLM（豆包视觉）识别
- **无 MD 中间文件**，直接入 `chroma_db`，通过终端日志实时审核
- 失败的 PDF 进 `Failed_PDFs/`，页面截图存 `slide_images/`
- 日志解析：`python parse_sync_log.py` 生成处理报告

审核用 **chunk_reviewer（8085）**，见第 4.2 节。

## 4. 审核工具详解

### 4.1 md_reviewer.py（端口 8084）— 常规文档 MD 审核
```bash
python md_reviewer.py
```
- 三列：源 PDF 全页自由滚动 | 编辑区 | 实时预览
- 用途：入库**前**校核 MD 内容、删目录、修表格、插图
- 见第 2 步骤 3

### 4.2 chunk_reviewer.py（端口 8085）— Chunk 数据质检
```bash
python chunk_reviewer.py
```
- 三栏：源文件（**可框选截图**）| 编辑 | 预览效果
- 用途：对**已入库**的块做质检/改写/配图（EHS案例、排雷库案例等）
- 截图存 `user_crops/`，索引 `user_crops/index.json`
- 块正文里 `[📷 路径]` 是图片标签；`[📷 待补]` 表示待补图
- 内置库外「回收站」：删除的块不会真没，可恢复
- 改动直接落地 ChromaDB（update 同 id + 重新编码向量）

## 5. 在线服务（机器人 / 网页）

### 5.1 飞书机器人（主力）
```bash
bash start_feishu.sh                  # 后台启动，自动重建 BM25
```
- 日志：`tail -f logs/feishu_ws_server_v11_$(date +%Y-%m-%d).log`
- PID 记录在 `feishu.pid`
- 停止：`kill $(cat feishu.pid)` 或 `pkill -f feishu_ws_server_v11`
- 检索链路：向量(ChromaDB) + BM25 → RRF 混合 → reranker 重排 → 喂豆包生成
- 问答日志存 `qa_logs/`（已加入 .gitignore，不上传）

### 5.2 网页聊天 UI（端口 8081）
需要先启动后端网关 8080：
```bash

python web_ui.py           # 网页 UI 8081
```
浏览器开 `http://127.0.0.1:8081`。

### 5.3 Telegram 机器人
```bash
python telegram_bot_v11.py
```
复用飞书的同一套检索核心，需要 `.env` 里配 `TELEGRAM_TOKEN`。

### 5.4 本地终端问答（调试用）

## 6. 实用操作

### 6.1 把 .doc 转成 .docx（rebuild_md 不吃 .doc）
LibreOffice 已安装（`soffice`）。转换并删除旧文件：
```bash
# 转某个 .doc（在同目录生成 .docx）
soffice --headless --convert-to docx --outdir "目标目录" "文件.doc"

# 批量转某文件夹下所有 .doc，成功后删原 .doc
find Local_KB/工管部 -type f \( -iname "*.doc" ! -iname "*.docx" \) -print0 | \
while IFS= read -r -d '' f; do
  soffice --headless --convert-to docx --outdir "$(dirname "$f")" "$f"
done
# 确认 .docx 都生成后再删：
find Local_KB/工管部 -type f \( -iname "*.doc" ! -iname "*.docx" \) -delete
```
> `.ods → .xlsx`、`.doc → .pdf` 同理，把 `docx` 换成目标格式。

### 6.2 重建 BM25 索引
最简单：重启飞书机器人（`bash start_feishu.sh`），启动时会自动重建。

### 6.3 跨版本迁移数据（如 v11 → v10）
ChromaDB 支持按 domain 过滤导出，向量可直接搬（v10/v11 都用 bge-m3，无需重新编码）。详见项目记忆，关键点：
- 同 source 先查目标库，有则先删再导（防重复）
- 搬完目标版本的 `bm25_index.pkl` 要重建

## 7. 备份 / 快照 / 恢复

### 7.1 改动前打快照（强烈建议）
```bash
python snapshot_kb.py
```
把 ChromaDB 当前所有块（带 id 原文）存成带日期的快照目录，作为人工改动前 before 基线。

### 7.2 数据库整库备份
当前已有的备份目录（手动 cp 生成）：
- `chroma_db.bak_20260529_203818/`
- `chroma_db.bak_20260601_全自动前/`

手动备份：
```bash
cp -r chroma_db "chroma_db.bak_$(date +%Y%m%d_%H%M%S)"
cp bm25_index.pkl "bm25_index.pkl.bak_$(date +%Y%m%d_%H%M%S)"
```

### 7.3 误删块恢复
- chunk_reviewer 内置库外「回收站」，删除的块存 `deleted_chunks/`，可恢复
- 更底层：可从 ChromaDB 的 `embeddings_queue` WAL 逐字+逐向量还原
- reviewer 内存视图过期时重启即可刷新

## 8. EHS 内容整改专用脚本（已是历史任务，了解即可）

这些是为 EHS 知识库内容整改写的一次性/半自动工具，大多 `dry-run`（不带 `--write` 只预览）：

| 脚本 | 用途 | 用法 |
|------|------|------|
| `denoise_ehs.py` | 删 EHS 块里页眉/版权/页码等噪音行 | `python denoise_ehs.py` 预览 / `--write` 写库 |
| `rework_ehs.py` | 按样板泛化整改同类块（删噪音+排版+补图标） | `python rework_ehs.py <source>` / `--write` |
| `ehs_autocrop.py` | 自动裁剪整排配图，复刻人工截法 | `python ehs_autocrop.py <source>` / `--write` |
| `reformat_cases.py` | 排雷库 V1/V2 案例重排成 V3.0 字段格式 | `python reformat_cases.py <source>` / `--write` |
| `rework_positive.py` | 正面·字段卡型块重排 | `python rework_positive.py <source>` / `--write` |
| `add_case_screenshots.py` | 自动裁剪案例右侧蓝框配图替换 `[📷待补]` | `python add_case_screenshots.py` |
| `parse_sync_log.py` | 解析 sync 日志生成报告 | `python parse_sync_log.py [日志路径]` |

> 整改原则：先 dry-run 看预览 → 确认无误再 `--write`。改前先 `snapshot_kb.py`。

## 10. 常见坑 & 排查

| 现象 | 原因 / 解决 |
|------|------------|
| `.doc` 文件没被解析 | rebuild_md 只吃 `.docx`，先转格式（6.1）|
| 表格在预览里没渲染成表格 | 表格前后要留空行；合并单元格要拆开重复填 |
| 改了库但飞书答案还是旧的 | 重启飞书机器人重建 BM25（`bash start_feishu.sh`）|
| 入库后某文件没进去 | index 有增量 skip；想强制重入用 `--force` 或先删旧块 |
| 8084/8085 打不开 | 看脚本是否在跑、端口是否被占；reviewer 内存视图过期重启 |
| EHS案例用错脚本 | 案例走 `sync_kb_v11.py`，常规文档走 `rebuild_md.py`，别搞反 |
| 想撤销一批改动 | 从 `chroma_db.bak_*` 整库恢复，或 chunk_reviewer 回收站 |
